# Отчет по лабораторной работе №8

## Тема

Изучение систем автоматизации развёртывания и управления приложениями на примере Docker.

## Цель работы

Научиться создавать Docker-образ для C++/CMake-проекта, собирать приложение внутри контейнера, запускать его с переменной окружения и сохранять результат работы через volume.

## Выполненные действия

1. За основу взят проект предыдущей лабораторной работы `lab07`.
2. Репозиторий подготовлен как `lab08`.
3. Добавлен файл `Dockerfile`.
4. В качестве базового образа выбран `ubuntu:18.04`, как указано в задании.
5. В контейнер устанавливаются инструменты сборки: `gcc`, `g++`, `cmake`, `make`, `git`, `ca-certificates`.
6. Исходный код копируется в каталог `/print` внутри контейнера.
7. Проект конфигурируется командой `cmake -S . -B _build`.
8. Проект собирается командой `cmake --build _build`.
9. Проект устанавливается в `/print/_install`.
10. Задана переменная окружения `LOG_PATH=/home/logs/log.txt`.
11. Добавлен volume `/home/logs` для сохранения логов вне контейнера.
12. Рабочая директория переключается на `/print/_install/bin`.
13. Точкой входа контейнера является приложение `demo`.
14. Добавлен `.dockerignore`, чтобы не копировать временные файлы сборки в Docker build context.
15. Вместо TravisCI настроен GitHub Actions workflow для сборки и проверки Docker-образа.

## Проверка локально

```bash
docker build -t logger .
mkdir -p logs
printf 'text1\ntext2\ntext3\n' | docker run --rm -i -v "$(pwd)/logs:/home/logs" logger
cat logs/log.txt
```

Ожидаемый результат: в файле `logs/log.txt` появляются строки `text1`, `text2`, `text3`, записанные приложением `demo`.

## Проверка в GitHub Actions

Workflow `.github/workflows/ci.yml` выполняет:

```bash
docker build -t logger .
printf 'text1\ntext2\ntext3\n' | docker run --rm -i -v "${{ github.workspace }}/logs:/home/logs" logger
```

Затем workflow проверяет наличие файла `logs/log.txt` и наличие всех трех тестовых строк.

## Вывод

В результате лабораторной работы был создан Docker-образ для C++/CMake-проекта. Сборка приложения выполняется внутри контейнера, запуск производится через `ENTRYPOINT`, а результат работы сохраняется на хост-машине через Docker volume.
